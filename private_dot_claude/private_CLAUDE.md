# Git Workflow Best Practices

## GitHub Flow

- Create a new branch for each feature/fix from `main`
- Branch naming: `<type>/<short-description>` (e.g., `feature/user-auth`, `fix/login-bug`)
- Open a Pull Request for code review before merging
- Merge to `main` only after PR approval
- Delete branch after merge

## Commits

- **Only commit when explicitly asked** - never commit automatically
- Make atomic commits according to the plan you set yourself
- Each commit should represent one logical change
- It can span multiple files if the step is relevant for all of them

## Commit Message Format

- **Single line only** - no multi-line messages, no body, no footers
- No "Generated by Claude" or "Co-Authored-By" lines

```
[tag] up to 6 words describing change
```

### Tags (one word representing context)
- `[feat]` - new feature
- `[fix]` - bug fix
- `[refactor]` - code refactoring
- `[docs]` - documentation
- `[test]` - tests
- `[style]` - formatting, no code change
- `[chore]` - maintenance tasks

### Examples
```
[feat] add user authentication endpoint
[fix] resolve login redirect loop
[refactor] extract validation helpers
[docs] update API usage examples
[test] add unit tests for parser
```

## Push Policy

- **Only push when explicitly asked** - never push automatically
- Never force push to `main` or `master`
- Never use `--force` unless explicitly requested

## Forbidden Actions

- Never use `git commit --amend` unless explicitly requested
- Never use `git rebase -i` (interactive)
- Never use `git reset --hard`
- Never skip hooks (`--no-verify`)
